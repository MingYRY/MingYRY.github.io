Elasticsearch

|字数总计:1.9k|阅读时长:5分钟|阅读量:

Elasticsearch

相比MySQL，能够自动帮我们做分词，能够非常高效、灵活地查询内容

索引

es中的索引 => MySQL 中的表

es中的字段（field）=> MySQL 中的列

正向索引：理解为书籍的目录，可以快速帮你找到对于的内容

倒排索引：根据文章内容里面的词汇来找到对应的文章

ES语法

DSL语言操作ES

不需要背

JSON格式的参数，和HTTP请求最兼容

EQL

不需要背

专门查询 ECS 文档（标准指标文档）的数据的语法，更加规范，特定场景使用

SQL语言操作ES

SQL能用尽量用，因为可以跨平台使用

学习成本低，可能需要额外插件的支持，性能较差

调用方式

restful api调用（HTTP请求）

get请求：http://localhost:9200/

curl 可以模拟发送请求： curl -X GET “localhost:9200/?pretty”

ES的启动端口

  1. 9200：给外部用户（客户端调用）的端口
  2. 9300：给ES集群内部通信的（外部调用不了）

kibana devtools

利用命令行 自由地对 ES 进行操作（本质也是 restful api）

dev tools 不建议生产环境使用

客户端调用

java 客户端操作 es：

  1. ES 官方的 Java API

  2. ES 以前的官方 Java API

    HighLevelRestClient（已废弃，不建议用）

  3. Spring Data Elasticsearch

    spring-data系列：spring 提供的操作数据库的框架

    spring-data-redis：操作 redis 的一套方法

    spring-data-mongodb：操作 mongodb 的一套方法

    spring-data-elasticsearch：操作 elasticsearch 的一套方法

    自定义查询方法：用户可以指定接口的方法名称，框架帮你自动生成查询

Mapping

类型MySQL中的表结构

ES 支持动态mapping，表结构可以动态改变

还支持显示创建mapping（手动指定mapping）

Runtime fields

这种字段属于运行时才确定值的字段值，可以保证动态计算（利用js的语法来进行计算）

如果存在该字段，就查找，如果不存在该字段，就不会查找

index 类型

  • text：进行全文搜索的类型

  • keyword：不可拆分，用于过滤和排序，关键字，用于结构化内容，如id、电子邮件地址、主机名、状态码、邮政编码或标签。 手机号和（身份证号因为有X）设置为keyword

  • boolean：布尔类型

  • numbers：数字的类型，

  • ip类型

  • range：范围类型

分词器

分词的一种规则

es中有很多内置的分词器：

​ 空格分词器：whitespace

​ 标准分词器：standard，将满足的条件进行过滤

​ 关键词分词器：就是不分词，整句话当作专业术语

IK 分词器（ES的插件）

中文友好

ik_smart，只能分词，尽量选择最项一个词的拆分方式

ik_max_word，尽可能地分词，最精细化进行分词，将“小黑子”，分成 “小黑”，“黑子”

打分机制

看内容的词语的匹配度，关注于匹配比例，es会进行打分

用 ES 实现搜索接口

1. 建表（建立索引）

ES 中，尽量存放需要用户筛选（搜索）的数据，频繁变化的数据也不适合放到ES中

根据数据库的表来建立 ES Mapping

ES Mapping：

id（可以不放到字段设置里）

aliases：别名（为了后续方便数据迁移），建表时尽量给别名

字段类型是text，这个字段是可被分词的、可模糊查询的；如果是keyword，只能完全匹配、精确查询

analyzer（存储时生效的分词器）：ik_max_word，拆的更碎、索引更多，更有可能被搜出来

search_analyzer（查询时生效的分词器）：ik_smart，更偏向于用户想搜的分词

如果想要让 text类型的分词字段也支持精确查询，可以创建 keyword 类型的子字段

ES 中 _字段表示系统的默认字段，比如 _id，如果系统不指定，会自动生成，但是不会再_source 字段中补充id的值，所以建议大家手动指定

2. 增删改查

第一种方式：ElasticsearchRepositiry，默认提供了简单的增删改查，多用于可预期的，相对不复杂的查询、自定义查询

第二种方式：Spring 默认给我们提供的操作 es 的客户端对象 ElasticsearchRestTemplate，也提供了增删改查，它的增删改查更灵活，适用于更复杂的操作，返回结果更完成，需要自己解析

查询 DSL语句：

  • bool：组合条件
  • must： 必须都满足
  • match：模糊查询
  • term：精确查询
  • range：范围查询
  • wildcard：模糊查询
  • regexp：正则匹配查询
  • score：代表查询出来的结果的分数
  • should：满足要求的个数才可以，minimum_should_match

3. 查询步骤：

  1. 获取参数
  2. 把参数组合为 ES 支持的搜索条件
  3. 从返回值中获取结果
  4. 先模糊筛选静态数据，查出数据后，在根据查到的内容id去数据库查找到动态数据

数据同步

一般情况下，如果做查询搜索功能；使用 ES 来模糊搜索，但是数据是存放在数据库 MySQL 为主，所以我们需要把MySQL 中的数据和 ES 进行同步，保证数据一致（以MySQL为主）。

首次安装完ES，把MySQL数据全量同步到ES里，写一个单词脚本

4种方式：全量同步（首次）+增量同步（新数据）

  1. 定时任务，比如一分钟一次，找到 MySQL 中过去几分钟（至少是定时周期的两到三倍）内发生改变的数据，然后更新到 ES。

    ES 根据id来更新或者查询数据

    优点：简单易懂、占用资源少、不用引入第三方中间件

    缺点：没有实时性

    应用场景：数据短时间内不同步影响不大、或者数据几乎不发生修改

  2. 双写：写数据库的时候，必须也去写ES；更新删除数据库同理。（事务：建议先保证MySQL写入成功，如果ES写失败了，通过定时任务+日志+告警进行检测和修复（补偿功能））

  3. 用 Logstash 数据同步和数据处理管道（一般要配合 kafka消息队列+ beats采集器）：

    配置config，整合了多种输入和输出的模式

    把MySQL 同步到 Elasticsearch

    增量配置：是不是可以只查最新更新的？可以记录上次更新的数据的时间

    全量更新：只需要将配置中的更新数据时间进行删除，然后第一次就会全量更新

    注意查询语句要按照updateTime进行排序，保证最后一条是最大的

    两个问题： 利用数据处理的filter组件

    • 字段全部变成小写了
    • 多了不想保存的数据

    优点：用起来方便，插件的

    缺点：学习成本高，一般需要配合其他的组件使用，比如kafka

  4. 订阅数据库流水的同步方式，canal（实时同步）

    原理：数据库每次修改时，会修改 binlog 文件，只要监听该文件的修改，就能第一时间得到消息并处理

    canal：帮你监听 binlog，并解析 binlog 为你可以理解的内容，伪装成mysql的从节点，获取主节点的 binlog

    优点：实时同步，实时性非常强

预编译SQL的优点：

  1. 灵活
  2. 模板好懂
  3. 部分防注入
  4. 主要是快（因为有缓存）
