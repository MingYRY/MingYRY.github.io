Java并发面试题

JUC常用的并发类

  1. atomic包

    CAS使用的原子操作类如，AtomicInteger等

  2. locks包

    AbstractQueuedSynchronizer (AQS)，ReentantLock（显式锁），ReentrantReadWriteLock (可重入的读写锁)

  3. tools包

    CountDownLatch（同步锁）、Semaphore（信号量）、CyclicBarrier（栅栏）

  4. collections包

    ConcurrentHashMap（同步HashMap）、CopyOnWriteArrayList（写时复制容器）、各种阻塞队列的实现：ArrayBlockingQueue（数组实现的有界阻塞队列，入队出队使用一把锁）、LinkedBlockingQueue（有界阻塞队列，入队出队两把锁）、PriorityBlockingQueue

  5. executor包

    Executor类，Future类，Callable类，RejcetedExecutionhandler（拒绝策略类）

condition在juc包中

自增操作的非原子性

  • 读取当前值
  • 增加当前值为新值
  • 写回新值

线程池的优势

  1. 降低系统资源消耗，通过重用已存在的线程，降低线程创建和销毁造成的消耗；
  2. 提高系统响应速度，当有任务到达时，无需等待新线程的创建便能立即执行；
  3. 方便线程并发数的管控，线程若是无限制的创建，不仅会额外消耗大量系统资源，更是占用过多资源而阻塞系统或oom等状况，从而降低系统的稳定性。线程池能有效管控线程，统一分配、调优，提供资源使用率；
  4. 更强大的功能，线程池提供了定时、定期以及可控线程数等功能的线程池，使用方便简单。

并发编程三要素

新建线程的几种方式

  1. 通过继承Thread类，重写run方法；
  2. 通过实现runable接口；
  3. 通过实现callable接口

线程的状态与转换

线程上下文切换的情况（4种）

  1. 时间片用完
  2. 调用sleep()，或者wait()
  3. 调用了阻塞类型的系统中断
  4. 被终止或结束运行

线程的几个方法（interrupt(), join(), sleep(), yield()）

  1. interrupt()，中断线程，来清理资源，相比之间结束线程更优雅

  2. join()，两个线程之间的通信，在A中执行B.join()，就是要等待B结束才能执行

  3. sleep()，线程休眠

    sleep() VS wait()

    两者主要的区别：

    1. sleep()方法是Thread的静态方法，而wait是Object实例方法
    2. wait()方法必须要在同步方法或者同步块中调用，也就是必须已经获得对象锁。而sleep()方法没有这个限制可以在任何地方种使用。另外，wait()方法会释放占有的对象锁，使得该线程进入等待池中，等待下一次获取资源。而sleep()方法只是会让出CPU并不会释放掉对象锁；
    3. sleep()方法在休眠时间达到后如果再次获得CPU时间片就会继续执行，而wait()方法必须等待Object.notift/Object.notifyAll通知后，才会离开等待池，并且再次获得CPU时间片才会继续执行。

  4. yield()，当前线程让出CPU，并且让出的时间片只会分配给当前线程相同优先级的线程

守护线程

用户线程消亡，守护线程也自动消失，jvm的垃圾清理线程

两个注意

  1. 守护线程在退出的时候并不会执行finnaly块中的代码，所以将释放资源等操作不要放在finnaly块中执行，这种操作是不安全的
  2. 设置守护线程要先于start()方法

JMM（Java内存区域）

结构模型

线程（本地变量（共享变量的副本））<—->主内存（共享变量）

通信过程

  1. 线程A读入共享变量，写到共享变量的副本中，修改共享变量的值，再写回共享变量中
  2. 线程B读取共享变量， 写到共享变量的副本中

重排序

为了提高性能，编译器和处理器常常会对指令进行重排序

  1. 编译器在不改变单线程语义的情况下，安排语句的执行顺序
  2. 处理器，如果不存在数据依赖性，会改变语句对应的机器指令的执行顺序

Happens-before（JMM原则就可以）

1）如果一个操作happens-before另一个操作，那么第一个操作的执行结果将对第二个操作可见，而且第一个操作的执行顺序排在第二个操作之前。

2）两个操作之间存在happens-before关系，并不意味着Java平台的具体实现必须要按照happens-before关系指定的顺序来执行。如果重排序之后的执行结果，与按happens-before关系来执行的结果一致，那么这种重排序并不非法（也就是说，JMM允许这种重排序）。

内存屏障

阻止指令重排序使用内存屏障

Volatile

Volatile特性

  1. 禁止指令重排序
  2. 保证变量的可见性

Volatile不适用场景

  1. 不适用于依赖先前状态的操作
  2. 不适用于不具备原子性的复合操作，例如自增
  3. 不适用于计数器
  4. 不适用于线程间复杂的通信

Volatile双重检验锁

Volatile使用场景

  1. 适用于变量状态的改变的场景

Synchronized

MarkWord

使用场景（两方面来回答）

  1. 方法上：
    • 静态方法：锁住的是整个类
    • 实例方法：锁住的是当前对象
  2. 代码块：
    • this：锁住的是当前对象
    • .class：锁住的是整个类
    • object：锁住的是目标对象

偏向锁（获取和撤销）

偏向锁获取：看当前的偏向锁的拥有线程是不是当前线程，如果是则再次获取偏向锁，如果不是，则CAS竞争锁

偏向锁撤销：当其他线程尝试竞争偏向锁时，持有偏向锁的线程才会释放锁。

偏向锁的撤销，需要等待全局安全点（在这个时间点上没有正在执行的字节码）。它会首先暂停拥有偏向锁的线程，然后检查持有偏向锁的线程是否活着，如果线程不处于活动状态，则将对象头设置成无锁状态；如果线程仍然活着，拥有偏向锁的栈会被执行，遍历偏向对象的锁记录，栈中的锁记录和对象头的Mark Word要么重新偏向于其他线程，要么恢复到无锁或者标记对象不适合作为偏向锁，最后唤醒暂停的线程。

轻量级锁（加锁和解锁）

加锁：然后线程尝试使用CAS将对象头中的Mark Word替换为指向锁记录的指针。如果成功，当前线程获得锁，如果失败，表示其他线程竞争锁，当前线程便尝试使用自旋来获取锁。

解锁：轻量级解锁时，会使用原子的CAS操作将Displaced Mark Word替换回到对象头，如果成功，则表示没有竞争发生。如果失败，表示当前锁存在竞争，锁就会膨胀成重量级锁。

普通的锁

Final

使用场景

类变量（static修饰）

  • 声明变量时直接赋初值
  • 静态代码块中给变量赋初值

实例变量

  • 声明变量时赋初值

  • 非静态初始块中

    就是{}中

  • 构造器中

赋值了之后不能再更改

局部变量

仅有一次机会被赋值，赋值后不能再次赋值

基本数据类型： 如果被赋初始值就不能再次赋值

引用数据类型：final只是保证引用的地址不变，对象的属性是可以改变的

方法

  • final修饰的父类不能被重写
  • final修饰的方法可以被重载

修饰不变类

不变类就是创建该类的实例后，该实例的实例变量是不可改变的（String）

  • 使用private和final来修饰成员变量
  • 使用带参数的构造器来初始化成员变量
  • 仅提供getter方法，普通方法无法修改final修饰的成员变量
  • 有必要的话，重写Object类的hashCode()方法和equals()方法

三大性质

原子性

一个操作不可中断，要么全部执行成功，要么全部执行失败

Volatile不能保证原子性除非两点满足：

  • 运算结果不依赖当前值，或者只有一个线程在操作
  • 变量不需要与其他的状态变量共同参与不变约束

有序性

在本线程内观察，所有操作都是有序的，但是在一个线程观察另一个线程，所有的操作都是无序的

Volatile的双重校验锁：

可见性

当一个线程修改了共享变量之后，其他线程能够立即得知这个修改。lock指令，在线程获取锁时从主内存中读取共享变量，释放锁时将共享变量写回到主内存中

Lock包

AQS（抽象队列同步器）

同步组件通过使用AQS提供的模板方法来实现同步组件的语义，AQS则负责对同步状态进行管理，对阻塞线程进行排队，等待通知等等一系列底层处理的实现。

AQS的核心：

  • 同步队列
  • 独占式锁的获取和释放
  • 共享锁的获取和释放
  • 可中断锁
  • 超时等待锁获取

同步队列

  1. 节点的数据结构（双向链表）

    有一个静态内部类Node，里面保存着当前节点的状态信息，前驱和后继的节点信息和当前节点对应的线程的引用

  2. 管理同步队列

    利用头尾指针来管理同步队列

独占锁的获取和释放

  • 线程获取锁失败，包装成一个Node进行入队操作，方法是 addWaiter() 和 enq()，并且使用enq()来完成队列的头结点初始化工作和CAS失败重试（当前节点加入队列的失败重试）
  • 获取锁是一个自旋等待的过程，当且仅当 当前节点的前驱节点是头结点，并且当前节点的状态值是同步状态时，该节点出队并且获取锁。不满足时就调用LockSupport.park()方法来进行线程阻塞
  • 释放锁之后会唤醒其后继节点。这就代表AQS是一个先入先出的同步阻塞队列

总体来说：在获取同步状态时，AQS维护一个同步队列，获取同步状态失败的线程会加入到队列中进行自旋；移除队列（或停止自旋）的条件是前驱节点是头结点并且成功获得了同步状态。在释放同步状态时，同步器会调用unparkSuccessor()方法唤醒后继节点。

带头节点的队列就是有一个虚拟头节点或者说是哨兵节点

ReentrantLock

可重入的互斥锁

特性：

  • 可重入式

    就是获取多少次锁，在state上自增，然后释放锁时，需要state为0

  • 公平锁和非公平锁

    • 公平锁保证请求资源时间上的绝对顺序，非公平锁可能导致其他线程无法获取到锁，造成饥饿的现象
    • 公平锁为保证时间上的顺序，频繁的进行上下文的切换。非公平锁一定程度上降低上下文切换的频率。ReentrantLock默认使用非公平锁

  • 等待可中断

Condition

Condition实现等待/通知的机制 配合Lock， 属于语言级别，有着更高的可控制性和扩展性

Object的wait/notify是利用对象监视器完成的线程等待/通知，属于Java底层级别

Codition的特性：

  • 支持不响应中断
  • 支持多个等待队列，Object只能支持一个
  • 支持超时时间的设置。

等待队列

是一个单向队列

一个Lock可以持有多个等待队列

await

当前线程调用condition.await()方法后，会使得当前线程释放lock然后加入到等待队列中，直至被signal/signalAll后会使得当前线程从等待队列中移至到同步队列中去，直到获得了lock后才会从await方法返回，或者在等待时被中断会做中断处理。

signal

调用condition的signal的前提条件是当前线程已经获取了lock，该方法会使得等待队列中的头节点即等待时间最长的那个节点移入到同步队列，而移入到同步队列后才有机会使得等待线程被唤醒，即从await方法中的LockSupport.park(this)方法中返回，从而才有机会使得调用await方法的线程成功退出。

signalAll

就是用一个循环全部唤醒

LockSupport

作用，阻塞当前线程

synchronzed致使线程阻塞，线程会进入到BLOCKED状态，而调用LockSupprt方法阻塞线程会致使线程进入到WAITING状态。

ConcurrentHashMap

初始大小为16并且每一次扩容后的大小是2的n次方

注意：调用构造器方法的时候，没有构造出table数组（这个数组是concurrentHashMap来存放数据的容器），只是算出了table的长度，等到插入数据时，才真正构造table数组

Node类：

  1. hash值
  2. key
  3. value
  4. next Node

Put方法

  1. 首先对Key的哈希值进行哈希计算，确定这个值在table（数组）中的位置
  2. 如果当前数组没有初始化，则进行初始化（插入第一个值时）
  3. 如果值所在数组位置的节点为null，则采用CAS的方式来放入key
  4. 如果节点不为空，代表出现了hash碰撞，首先需要判断节点的类型。如果fh==-1，表示正在扩容，则参与到扩容中
  5. 如果是链表节点，则需要遍历链表，如果key值相同，则覆盖旧值，如果不存在相同的key值，则尾插法插入到链表的尾部（synchronized）
  6. 如果节点是TreeBin的话，遍历红黑树，相同key值覆盖，不相同则插入树中（synchronized）
  7. 插入完节点后，检查链表的长度，如果大于8则再判断数组长度是否小于64，如果小于64，则对数组扩容。如果大于64，则将链表转换为红黑树
  8. 对当前的容量大小进行检查，如果超过了临界值（实际大小*加载因子（3/4））就需要扩容

Get方法

  1. 查看当前数组的节点是否是要查找的节点，是就返回
  2. 不是，则判断当前的数组节点是否是红黑树节点，是的话查找树，查到返回值，查不到返回null
  3. 不是红黑树节点，则是链表节点，进行链表的查找，查到返回值，查不到返回null

JDK1.7 和 JDK1.8的区别

JDK1.7

采用Segment（哈希桶）+ HashEntry的方式进行实现

Segment默认大小为16，HashEntry默认大小为2的幂次方。

加锁是对segment的每个节点加锁（加ReentreantLock锁），如果数组的位置对应节点没有被初始化，则使用CAS进行初始化

ThreadLocal

线程的本地变量。即每个线程都拥有该变量的副本，避免共享资源的竞争（怎么判断是对应线程的变量？）

ThreadLocalMap

里面维护了一个Entry的数组

Entry

是一个以ThreadLocal为Key，Object为值的键值对。其中，ThreadLocal是一个弱引用

Entry中的key是弱引用，当threadLocal外部强引用被置为null( threadLocalInstance=null ),那么系统 GC的时候，根据可达性分析，这个threadLocal实例就没有任何一条链路能够引用到它，这个ThreadLocal势必会被回收，这样一来，ThreadLocalMap中就会出key为null的Entry，就没有办法访问这些key为null的Entry的value，如果当前线程再迟迟不结束的话，这些key为null的Entry的value就会一直存在一条强引用链Thread Ref -> Thread -> ThreaLocalMap -> Entry -> value永远无法回收，造成内存泄漏。

在实际开发中，会使用线程池去维护线程的创建和复用，比如固定大小的线程池，线程为了复用是不会主动结束的，所以，threadLocal的内存泄漏问题，是应该值得我们思考和注意的问题。

ThreadLocal的生命周期中会清除key为null的脏Entry

解决哈希冲突

使用开放地址法，原因，ThreadLocalMap中的散列值的分布十分均匀，很少出现冲突，并且ThreadLocalMap经常清除无用对象，使用数组更加方便

threadLocal最佳实践

我们可以完全理解threadLocal内存泄漏的前因后果，那么实践中我们应该怎么做？

  1. 每次使用完ThreadLocal，都调用它的remove()方法，清除数据。
  2. 在使用线程池的情况下，没有及时清理ThreadLocal，不仅是内存泄漏的问题，更严重的是可能导致业务逻辑出现问题。所以，使用ThreadLocal就跟加锁完要解锁一样，用完就清理。

threadLocal只适合共享对象会造成线程安全的业务场景

BlockingQueue 阻塞队列

特殊操作：

  1. put：当阻塞队列容量已满时，往阻塞队列里面插入数据会被阻塞，知道有空余容量可以使用

  2. offer(E e, long timeout, TimeUnit unit)：与put相同，但是可以设置超时时间，超过超时时间，线程会退出

  3. take()：当阻塞队列为空时，获取队头数据的线程会被阻塞

  4. poll(E e, long timeout, TimeUnit unit)：和offer差不多
